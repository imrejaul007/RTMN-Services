# Rendez — Admin Guide

Admin dashboard lives at `https://rendez-admin.vercel.app` (or `http://localhost:3000` locally).

All requests to the backend use `Authorization: Bearer <ADMIN_API_KEY>`.

---

## Dashboard

**Route:** `/dashboard`

Shows platform health at a glance:
- **KPI cards:** Total Users, Active Matches, Gifts Sent, Validated Meetups
- **Line chart:** Daily breakdown of matches / gifts / meetups / new users
  - Range selector: 7d / 14d / 30d
  - Toggle individual series on/off
- **Pending alerts:** Pending Reports, Fraud Flags (unresolved), Active Gifts

---

## Users

**Route:** `/users`

| Feature | Notes |
|---------|-------|
| Search | By name (live, debounced) |
| Filters | City, verified status, active status |
| Suspend | Immediate — user gets 403 on next request |
| Unsuspend | Restores access instantly |
| isSuspended badge | Red pill on row; row dimmed |

**Suspend behavior:** Sets `isActive=false, isSuspended=true` on the Profile record. The `rendezAuth` middleware checks `isSuspended` on every authenticated request, so effect is immediate — no token invalidation needed.

---

## Gifts

**Route:** `/gifts`

- 4 KPI cards: Total, Active (pending), Value Accepted (₹), Acceptance Rate
- Status breakdown mini-bars (PENDING / ACCEPTED / REJECTED / REDEEMED / EXPIRED / CANCELLED)
- Gift type split (COIN vs MERCHANT_VOUCHER)
- Conic-gradient donut chart for acceptance rate
- Paginated table (20/page): sender, receiver, type, amount, status, date

---

## Meetups

**Route:** `/meetups`

- 4 KPI cards: Total Meetups, Validated (both checked in), Rewards Triggered, Failed Rewards
- Top Venues bar chart: top 10 merchants by meetup count
- Filter tabs: ALL / TRIGGERED / PENDING / FAILED
- Table: match users, booking ID, check-in count (x/2), check-in date, reward status, reward ref

---

## Moderation

**Route:** `/moderation`

Report queue — all user-submitted reports.

- Filter tabs: PENDING / REVIEWED / ACTION_TAKEN / DISMISSED
- Each row: reporter, reported user + city, reason badge (color-coded), date
- **Resolve** → sets status to `ACTION_TAKEN`
- **Dismiss** → sets status to `DISMISSED`

After resolving, follow up in the Users page to suspend the offending account if warranted.

---

## Fraud

**Route:** `/fraud`

Automated fraud flags generated by backend fraud sweeps.

| Flag Type | Trigger |
|-----------|---------|
| GIFT_SPAM | >3 gifts/day to same receiver |
| REWARD_FARMING | Reward attempt on match <7 days old or pair <90 days since last reward |
| MULTIPLE_ACCOUNTS | Same phone registered >1 profile |
| FAKE_CHECKIN | Check-in without valid booking in Redis cache |

- Toggle: Active / Resolved
- Resolve button marks flag as `resolved=true`
- Color-coded badges per flag type

---

## API Authentication

Every admin page fetches from `${NEXT_PUBLIC_API_URL}/admin/...` with:
```
Authorization: Bearer <ADMIN_API_KEY>
```

If the backend returns `401` or `403`, the API key is missing or wrong. Verify `ADMIN_API_KEY` matches in both the backend env and the admin env.

---

## Common Operations

### Suspend a user after a report
1. Moderation → find report → Resolve
2. Users → search reported user → Suspend

### Investigate reward farming
1. Fraud → filter by REWARD_FARMING
2. Note the userId and match details
3. Users → find user → review profile
4. If confirmed: Suspend

### Check if a meetup reward fired
1. Meetups → search by user name or scroll to date
2. Check Reward column (TRIGGERED = coins sent, FAILED = REZ error, PENDING = waiting)
3. Reward Ref column shows REZ transaction reference for escalation to REZ partner support
